06 / PRESENTER GLOSSARY

Plain-language glossary

Every specialized label used above is translated here into what it is and why this system needs it.

Actionable event
A log event with enough evidence to investigate, kept so incomplete or harmless noise does not enter the fixing process.
Adapter
A small translator between one app and the shared pipeline, used so each app can trigger its bugs in its own way.
Agent
An AI worker given one bounded job, used where judgment is needed and checked before the run advances.
Assertion spec
A written list of facts that must be true after a fix, used to turn expected behavior into checks the verifier can run.
Checkpointer
A component that saves a known-good snapshot after each step, used so an interrupted run can continue without starting over.
Claude Agent SDK
A library for giving Claude bounded tools and requiring structured answers, used by triage to inspect evidence without editing code.
Command-line tool
A program controlled with typed commands, used so the pipeline can start tools and capture their results automatically.
Conditional edges
Graph connections chosen by a yes/no result, used to retry a failed check or advance a passing fix.
Contract
An agreed data shape and rule between steps, used so both implementations can exchange information safely.
Dedupe
The step that recognizes repeated reports of the same failure, used to reuse one issue instead of opening duplicates.
Deterministic code
Code that follows fixed rules and gives the same result for the same input, used to keep routine work predictable.
DRY_RUN
A safety switch that shows intended writes without changing GitHub or Git, used for safe rehearsals.
Fingerprint
A repeatable short ID made from a failure’s important details, used to recognize the same bug across many log entries.
Fixer interface
The shared input and output expected from a fixing tool, used so Grok or Codex can be swapped without changing the pipeline.
Framework
Reusable plumbing that organizes the steps, used here to compare graph-based control with ordinary code.
Git
The system that records code history and separate branches, used here to isolate each attempted fix and inspect exactly what changed.
GitHub
The shared service that stores this project’s issues and pull requests, used to hand verified work to people for review.
Harness
The surrounding tool that starts an AI worker, limits its capabilities, and records its result, used to run different workers consistently.
Heuristic classifier
A short list of practical rules that separates likely bugs from routine noise, used so only useful events continue.
Incident
One distinct bug tracked through a run, used to keep its evidence, fix attempts, and ticket together.
JSONL
A text format with one complete data record per line, used so new log entries can be read without loading the whole file.
LangGraph
A framework that represents work as connected steps, used to make saved progress and branching choices visible.
Log bookmark
A saved read-position that makes reruns process only new log lines, stored by the code as .cursor.json through cursorPath.
MemorySaver
LangGraph’s in-memory checkpointer, used here to retain the current run state between graph steps.
Model
The specific AI system doing a judgment task, named so each run records which intelligence produced the decision.
Orchestration
The code that decides which step runs next, used so agents cannot choose their own permissions or skip checks.
Pipeline
The fixed sequence of steps that turns raw logs into checked fixes, used to enforce one safe route from evidence to action.
Pull request / PR
A request to review and merge a proposed code change, opened only after this system’s independent checks pass.
Repro / reproduction
A reliable way to make the bug fail on demand before editing code, used to prove that the later fix solved the real problem.
red→green
The change from a failing check before the fix to a passing check afterward, used as direct evidence that behavior improved.
Resumability
The ability to continue from the last saved step after an interruption, used to avoid repeating completed work.
Root cause
The underlying defect that created the visible symptom, targeted so the fix prevents the failure instead of hiding it.
Scope guard
A rule that blocks reads or writes outside approved paths, used to keep investigation and fixes inside the intended code area.
Seam
A deliberate boundary where one implementation can be replaced by another, used to swap tools without disturbing the surrounding system.
StateGraph
LangGraph’s map of steps and connections, used to make every allowed path through a run explicit.
Telemetry
The measurements and event history emitted by a run, kept so people can inspect what happened instead of trusting a black box.
Trace
The ordered record of what each step did and why, kept so a run can be inspected or replayed later.
Triage
The decision step that reads evidence, judges whether automation is safe, and writes a focused plan for the fixer.
Typecheck
A check that TypeScript values are used in compatible ways, run to catch broken code paths before a fix advances.
TypeScript
JavaScript with checked data shapes, used so pipeline steps agree on the information they exchange.
Verifier
The independent gate that reruns the bug proof and code checks, used so an agent cannot approve its own work.
Worktree
A separate working copy of a Git project, used so an attempted fix cannot disturb the main project folder.
